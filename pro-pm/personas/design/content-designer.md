# Content Designer

## Role Overview

Owns UX writing, interface copy, terminology consistency, content strategy, and localization readiness. Responsible for ensuring every word in the product is clear, helpful, and consistent. Words shape how users understand and navigate the product — and bad copy causes confusion, support tickets, and lost trust. Success means users read, understand, and act confidently at every step.

## What They Care About Most

1. **Clarity over cleverness** — Users are scanning, not reading. Every word must earn its place. Jargon, ambiguity, and marketing-speak in UI cause confusion.
2. **Terminology consistency** — If it's called a "workspace" in one place and a "project" in another, users lose trust. Consistent vocabulary matters enormously.
3. **Error and guidance copy** — Error messages and help text are the most important copy in the product. They write the words users see when they're frustrated or confused.
4. **Localization readiness** — UI copy that works in English but breaks in translation creates global product debt. Sentence structure, string length, and cultural context all matter.
5. **Voice and tone** — The product should sound like one person. Consistent voice builds brand and trust; tonal variation matches the context (celebratory for success, calm for errors).

## How They Think

Thinks in user comprehension: "Will the user understand this word? Will they know what to do next? Is this copy doing a job, or is it just filling space?" Evaluates proposals by reading through every user-facing string with fresh eyes. Tests copy against edge cases — long names, error states, first-time users, non-native speakers. Considers the full content ecosystem — onboarding, tooltips, emails, docs — not just individual screens.

## Communication Style

Wants to see full user flows with context, not isolated strings to wordsmith. Prefers to be involved in design from the beginning, when content structure decisions are made. Appreciates when PMs share user research and support data that reveals where users are confused. Dislikes being asked to "just write the button text" without understanding the flow.

## Common Concerns When Reviewing PM Proposals

- "The terminology in this spec doesn't match what we use elsewhere in the product"
- "These error messages don't tell the user what to do next"
- "This copy won't localize well — the sentence structure is too English-specific"
- "The feature naming is driven by internal terminology, not user language"
- "There's no content plan for onboarding, tooltips, or help text"

## How to Get Their Buy-In

Include content design in the design process from the start — not at the end for a copy pass. Share the user context so they can write copy that meets users where they are. Be open to renaming features if user research shows the current name causes confusion. Treat localization as a requirement, not a future enhancement.

## Red Flags They Watch For

- Product names or features using internal jargon that users don't understand
- Placeholder copy ("Lorem ipsum" or developer-written strings) shipping to production
- Error messages that say what went wrong but not how to fix it
- Inconsistent terminology across features, platforms, or documentation
- Content decisions made without considering translation and localization impact
- Copy treated as decoration rather than a functional design element

## Relationship with Product

Values PMs who recognize that words are part of the user experience, not an afterthought. Frustrated when content design is brought in late to "polish" copy that was never well-structured. Best collaboration happens when PMs involve content design in naming decisions, user flow design, and terminology standards from the beginning.

## Key Questions They Will Ask

1. "Is this terminology consistent with how we name similar concepts elsewhere?"
2. "What does the user need to understand and do at this step?"
3. "What are the error states, and what do we tell the user in each one?"
4. "Is this feature name meaningful to users, or is it internal language?"
5. "What's the localization plan — how many languages, and are there string length constraints?"
6. "Where does onboarding, tooltip, and help content fit into this feature?"
