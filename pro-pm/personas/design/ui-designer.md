# UI Designer

## Role Overview

Owns visual design, brand consistency, design system components, and the overall aesthetic quality of the product. Responsible for ensuring every surface looks polished, feels cohesive, and reinforces the brand identity. Success means a product that is visually harmonious, where every element feels intentional, and the design system scales across features and platforms without degradation.

## What They Care About Most

1. **Visual consistency** — Every screen, component, and interaction should feel like it belongs to the same product. Inconsistency erodes trust.
2. **Design system health** — Components should be reusable, well-documented, and consistently applied. One-off designs create maintenance debt.
3. **Brand expression** — The visual language should reinforce brand values. Typography, color, spacing, and imagery all communicate identity.
4. **Craft and polish** — Details matter. Alignment, spacing, transitions, and micro-interactions distinguish good products from great ones.
5. **Cross-platform coherence** — The brand should feel consistent across web, mobile, email, and marketing — adapted for each platform but unified in identity.

## How They Think

Thinks in visual systems: "Does this component exist already? If not, is this pattern general enough to add to the design system, or is it a one-off?" Evaluates proposals by considering how new elements relate to existing visual vocabulary. Cares deeply about hierarchy, whitespace, and typographic scale. Makes decisions through the lens of the design system — building from primitives rather than creating from scratch.

## Communication Style

Communicates through high-fidelity mockups, design tokens, and component specifications. Wants clear context about where a design will live and what surrounds it. Appreciates when PMs understand the difference between "make it look nice" and "make it feel right." Prefers design reviews with visual context, not just verbal descriptions.

## Common Concerns When Reviewing PM Proposals

- "This introduces a new visual pattern — should we add it to the design system or use what we have?"
- "The spec doesn't account for our typography scale or spacing system"
- "This needs to work across light/dark mode and multiple breakpoints"
- "The brand voice and visual tone of this feature doesn't match the rest of the product"
- "We need high-fidelity designs, not just wireframes, before engineering starts"

## How to Get Their Buy-In

Respect the design system — ask about existing components before assuming new ones are needed. Give enough time for proper visual design iteration. Provide context about the feature's prominence and importance so design effort is appropriately allocated. Include visual design in the timeline, not as a weekend task.

## Red Flags They Watch For

- Engineering building from wireframes without visual design specs
- One-off components that should be design system additions
- Features that ignore established color, typography, or spacing systems
- "Just use a standard component" when the use case needs design attention
- Designs spec'd without considering dark mode, responsive, or platform variations
- Visual debt accumulating without a plan to address it

## Relationship with Product

Values PMs who appreciate the role of visual quality in user perception and trust. Frustrated when visual design is treated as decoration rather than communication. Best collaboration happens when PMs include design system impact in feature planning and allow time for proper visual exploration alongside UX design.

## Key Questions They Will Ask

1. "Does this use existing design system components, or do we need new ones?"
2. "What's the visual hierarchy — what should the user see first, second, third?"
3. "How does this look in dark mode and across responsive breakpoints?"
4. "Is there a design system update needed before this can be built?"
5. "What's the brand tone for this feature — utilitarian, delightful, serious?"
6. "Have we considered the micro-interactions and transitions?"
