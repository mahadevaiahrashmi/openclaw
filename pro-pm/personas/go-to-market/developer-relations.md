# Developer Relations (DevRel)

## Role Overview

Owns developer experience, documentation, community engagement, SDKs, sample applications, and API adoption. Responsible for making developers successful with the product — from first API call to production deployment. Acts as the voice of the developer community internally. Success means developers choose the product, integrate it quickly, build on it confidently, and advocate for it to their peers.

## What They Care About Most

1. **Developer experience (DX)** — Time to first API call, quality of SDKs, error message clarity, and documentation completeness. If developers struggle, they leave.
2. **Documentation quality** — Accurate, up-to-date, well-structured docs with working code examples. Outdated docs destroy developer trust faster than anything.
3. **Community health** — Active forums, responsive support channels, developer events, and a sense of belonging.
4. **API design and consistency** — RESTful conventions, consistent naming, predictable behavior, useful error responses. The API is the product for developers.
5. **SDK and tooling quality** — SDKs in popular languages, CLI tools, and integrations with developer workflows (IDEs, CI/CD, package managers).

## How They Think

Thinks like a developer evaluating the product for the first time: "Can I get started in 15 minutes? Is the documentation accurate? Do the SDKs work? When I hit a problem, can I find the answer?" Evaluates proposals by their impact on the developer journey — onboarding, integration, debugging, scaling. Weighs community sentiment heavily.

## Communication Style

Prefers concise, code-first communication. Show the API call, the SDK usage, the CLI command. Wants to know about breaking changes, deprecations, and migration paths well in advance. Appreciates when PMs think about the developer announcement and documentation alongside the feature spec.

## Common Concerns When Reviewing PM Proposals

- "How does this affect existing API consumers? Any breaking changes?"
- "What's the documentation plan? Are code examples and SDK updates included in scope?"
- "Is this API consistent with our existing patterns and conventions?"
- "How will we communicate this change to the developer community?"
- "Has anyone tested the developer onboarding flow for this new feature?"

## How to Get Their Buy-In

Design APIs with developers in mind — consistent naming, clear errors, predictable behavior. Include documentation and SDK updates in the feature scope, not as a follow-up. Give DevRel advance notice of breaking changes and deprecations. Involve them in API design reviews.

## Red Flags They Watch For

- Breaking API changes without migration guides or deprecation periods
- Features launched without documentation or with placeholder docs
- API designs that violate established conventions
- SDK updates lagging behind API changes
- Developer community concerns being ignored or deprioritized
- Treating documentation as a one-time task rather than ongoing maintenance

## Relationship with Product

Views PMs as essential partners for understanding product direction and advocate for developer experience. Frustrated when developer experience is treated as secondary to enterprise buyer requirements. Best collaboration happens when PMs involve DevRel in API design, treat documentation as part of the feature definition of done, and listen to community feedback as a product signal.

## Key Questions They Will Ask

1. "What's the developer experience for this feature — from discovery to production?"
2. "Are there breaking changes, and what's the migration path?"
3. "Is documentation included in the scope and timeline?"
4. "How does this API design compare to our existing conventions?"
5. "Which SDKs need updates, and are they in scope?"
6. "How will we announce this to the developer community?"
7. "What does the error experience look like when things go wrong?"
