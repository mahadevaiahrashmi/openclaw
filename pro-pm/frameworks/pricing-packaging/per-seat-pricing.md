# Per-Seat Pricing

## What It Is
Per-seat (per-user) pricing charges a fixed amount for each user who accesses the product. It's the most common SaaS pricing model because it's simple, predictable, and scales with organizational adoption. Variants include per-active-user (only counting users who log in), per-named-user, and tiered per-seat (different prices for different user types).

## When to Use
- Collaboration tools where value increases with more users
- Products where each user has a distinct account and usage pattern
- When you want predictable, scalable revenue that grows with customer organizations
- B2B SaaS where procurement teams need straightforward per-unit costs

## When NOT to Use
- Products used by one person who serves many (analytics dashboards viewed by the org)
- When per-seat creates "seat hoarding" or "shadow IT" problems (users sharing logins)
- Products where value is driven by usage volume, not user count
- Developer tools where one integration serves the whole team

## How to Apply
1. **Define "seat"**: Named user, active user (logged in within 30 days), or concurrent user?
2. **Set the price per seat**: Based on value-based or competitive pricing analysis
3. **Consider user tiers**: Admin ($X), Power User ($Y), Viewer ($Z) — align price to value received
4. **Volume discounts**: Encourage larger deployments. Example: $15/seat for 1–50, $12/seat for 51–200
5. **Minimum commitment**: Require a minimum seat count for certain tiers (e.g., Enterprise requires 100+)
6. **Growth mechanics**: Make it easy to add seats. Alert admins when approaching limits. Offer bulk discounts

## Example
**Product**: Team communication platform
- **Standard User**: $8/seat/month (messaging, channels, search)
- **Power User**: $15/seat/month (analytics, integrations, compliance features)
- **Guest/Viewer**: Free (read-only access, encourages network expansion)
- **Volume**: 100+ seats → 15% discount. 500+ → 25% discount
- **Growth**: Free viewer accounts create viral adoption; viewers convert to paid when they need to post/react

## Key Pitfalls
- Seat counting friction — if adding seats requires procurement approval, adoption slows
- Shadow IT — when seats are expensive, teams share logins, reducing your visibility and security
- Penalizing adoption — if adding users increases cost significantly, champions hesitate to expand
- Not differentiating user types — a CEO who logs in once a month shouldn't cost the same as a power user
- Undercounting value — for products where one user generates value for many (e.g., analyst serving 50 viewers)

## Related Frameworks
- **Usage-Based Pricing** — alternative when value correlates with consumption, not user count
- **Tiered Packaging** — per-seat pricing works within Good/Better/Best tiers
- **Freemium Model** — free viewer/guest seats as a growth mechanism
- **Expansion Revenue** — seat expansion is a primary driver of NRR
